# Verification Signal Study: 综合 Insight 分析

> **实验日期**: 2026-03-11
> **研究主题**: 验证信号密度与类型如何影响 AI Agent 在不同领域任务中的表现
> **两个基准**: NL2RepoBench (软件工程) + NewtonBench (科学发现)

---

## 1. 核心结论

### **密集定量反馈（C arm）在两个截然不同的领域均为最优策略**

| 维度 | NL2Repo (软件工程) | NewtonBench (科学发现) |
|------|-------------------|----------------------|
| C arm 的核心指标 | heldout ratio **36.1%** (最高) | avg best R² **0.634** (最高) |
| C arm 的 wins | **4.83** / 13 任务 | **8.0** / 19 任务 |
| C arm 的 strict wins | **2** | **6** |
| 第二名 | B_staged (4.33 wins) | A_batched (5.0 wins) |
| 最差表现 | A_sparse (1.83 wins) | B_interactive (3.0 wins) |

## 2. 四组实验的通俗解读

### 2.1 NL2Repo（软件工程）：你能看到多少测试结果？

任务是"从自然语言描述生成一个完整的代码仓库"。四组的区别在于 Agent 写代码过程中**能看到多少隐藏测试的结果**：

| 组 | 通俗比喻 | 具体做法 |
|---|---|---|
| **A - Sparse Final Only** | 闭卷考试 | 整个写代码过程中**看不到任何测试结果**，写完交卷才一次性跑全部测试。盲写代码。 |
| **B - Staged Modules** | 阶段考试 | 每完成一个模块阶段，才看到**该模块组**的测试结果。写完一章老师批改一章。 |
| **C - Dense Any Test** | 随时开卷 | **随时可以运行**所有非 heldout 测试，想验证几次验证几次。边写边跑测试。 |
| **D - Progressive Release** | 逐步揭秘 | 按进度解锁：0%→只看 import 测试，30%→加 interface，60%→加 functional，80%→加 integration。 |

最终评分：所有组都用一组 Agent **从未见过的 heldout 测试**（占 20%）来打分，确保公平。

### 2.2 NewtonBench（科学发现）：你能知道自己的假设有多对？

任务是"通过设计实验发现物理定律"（如发现万有引力公式 F=Gm₁m₂/r²）。四组的区别在于 Agent 提出假设后**能获得什么类型的反馈**：

| 组 | 通俗比喻 | 具体做法 |
|---|---|---|
| **A - Batched Dataset** | 一次采集 | 一次花掉全部实验预算（10次），拿到所有原始数据后**只能分析，不能再做实验**。不告诉假设对不对。 |
| **B - Interactive Observation** | 正常科研 | 做一个实验→看结果→调整假设→再做实验。每次只看到**原始观测数据**，不告诉假设的拟合度。 |
| **C - Quantitative Feedback** | 带评分的科研 | 和 B 一样可交互实验，但每次提假设后系统告诉你：**"你的 R² 是 0.73，RMSLE 是 0.15"**。知道"错了多少"。 |
| **D - Directional Feedback** | 带指导的科研 | 和 B 一样可交互实验，但每次提假设后系统告诉你：**"你在 distance 高值区间高估了约 35%"**。知道"哪里错了"。 |

最终评分：所有组都在 Agent 从未见过的隐藏验证点上计算 R²。

### 2.3 四组的本质梯度

```
信息量:  无反馈 ──────────────────────────→ 最密集定量反馈

NL2Repo:    A(闭卷)  →  B(阶段批改)  →  D(逐步揭秘)  →  C(随时开卷)
NewtonBench: A(只给数据) → B(可交互)  →  D(告诉方向)  →  C(告诉分数)
```

结论：**C 组（最密集的定量反馈）在两个领域中都表现最好**。

## 3. 跨领域共性 Insight (4 条)

### Insight 1: 信号密度正相关于任务成功率

两个基准一致表明：**Agent 获得的反馈越密集、越具体，最终表现越好**。

- **NL2Repo**: C (dense) >> D (progressive) > B (staged) > A (sparse)
- **NewtonBench**: C (quantitative) >> A (batched) > D (directional) > B (interactive)

稀疏反馈（A arm）在两个领域中都不是最优的，即使它是"最接近官方 benchmark 行为"的策略。

### Insight 2: 密集反馈在中等难度任务上效果最显著

反馈密度的价值**不是在极简单或极难的任务上体现**，而是在"有可能做对、但需要反馈引导"的中等任务上体现。

**NL2Repo 证据**:
- 简单任务 (retrying, 23 tests): 所有 arm 都能通过 ~22/23，差异不大
- 中等任务 (aiofiles, 202 tests): C arm 从 0 跃升到 **151/194/43**，其他 arm 基本为 0
- 困难任务 (autopep8, dictdatabase 等): 所有 arm 均为 0

**NewtonBench 证据**:
- 简单定律 (gravity, coulomb 等): 所有 arm 都 R²=1.0
- 中等定律 (snell, malus, be_distribution): C arm **独占优势** (snell: 0→1.0, malus: 0.41→1.0)
- 困难系统 (simple_system + snell): 所有 arm 均为 0

### Insight 3: 方向性反馈 ≠ 定量反馈

D arm (directional/progressive) 虽然也提供了额外信息，但效果显著弱于 C arm (quantitative/dense)：

| 对比 | NL2Repo | NewtonBench |
|------|---------|-------------|
| C wins | 4.83 | **8.0** |
| D wins | 2.00 | 3.0 |
| C avg metric | 36.1% heldout ratio | 0.634 R² |
| D avg metric | 22.0% heldout ratio | 0.562 R² |

这表明 **"知道自己哪里错了多少" 远比 "知道自己哪个方向错了" 更有价值**。

### Insight 4: 信号类型与领域特性的交互效应

虽然 C arm 整体最优，但两个领域的"第二名"不同，揭示了领域特性的影响：

- **NL2Repo (软件工程)**: B_staged (4.33) > D_progressive (2.00)
  - 软件工程中，**结构化的阶段性反馈**（按模块验证）比渐进式解锁更有效
  - 代码有明确的模块边界，阶段性验证与软件开发的自然节奏匹配

- **NewtonBench (科学发现)**: A_batched (5.0) > D_directional (3.0) > B_interactive (3.0)
  - 科学发现中，**一次看到全部数据**反而是较好的策略
  - 科学推理需要 holistic pattern recognition，数据越多越有利于发现规律

## 5. 对实验设计的反思

### 4.1 优势

1. **无 data leakage**: NL2Repo 的 heldout 测试从未在 Agent 运行期间暴露；NewtonBench 的评估点由模块内部随机生成
2. **控制变量合理**: 所有 arm 使用相同的 Agent、相同的 budget，仅改变反馈信号
3. **跨领域验证**: 两个截然不同的领域得出一致结论，增强结论的可靠性

### 4.2 局限性

1. **样本量较小**: 仅 20 个 pilot 任务，需要在完整 104 / 324 任务上验证
2. **单次运行**: 每个 (task, arm) 组合只跑了一次，无法估计方差
3. **仅 easy 难度**: NewtonBench 仅测试了 easy 难度，medium/hard 上的结论可能不同
4. **任务成功率偏低**: NL2Repo 中 11/20 任务全部为 0，限制了统计分析的 power

### 4.3 后续建议

1. **扩大样本量**: 在完整任务集上运行
2. **增加重复**: 至少 3 次重复以估计方差
3. **探索 medium/hard**: NewtonBench 扩展到更高难度
4. **混合策略**: 尝试 C+D 的组合（定量 + 方向性反馈同时给出）
5. **信号粒度研究**: 在 "dense" 和 "sparse" 之间找到最优反馈频率

## 6. 理论解读

### 5.1 从 RL 角度

密集反馈相当于提供了**更密集的 reward signal**，使 Agent 的搜索更高效：
- Sparse (A): 类似 episode-level reward，credit assignment 困难
- Dense (C): 类似 step-level reward，每一步都能获得梯度方向

### 5.2 从认知科学角度

密集定量反馈实质上为 Agent 提供了**外部化的工作记忆扩展**：
- Agent 本身的 reasoning 有限
- 定量反馈帮助 Agent 将"我的假设有多好"这一关键信息从模糊推理外化为精确数字
- 方向性反馈（D）提供了方向但不精确，效果介于 B 和 C 之间

### 5.3 实践意义

**在设计 AI Agent 的工作流时，应尽可能提供密集、定量的验证反馈**：
- 软件工程: 频繁运行测试、提供具体的 pass/fail 信息
- 科学发现: 提供 R²、误差等具体数值，而非仅告知"方向"
- 通用: "Tell the agent exactly how wrong it is" 比 "Tell the agent which direction to go" 更有效

## 7. 从实验室到现实：没有 Ground Truth 时怎么办？

本实验的一个根本前提是：我们**知道正确答案**（NL2Repo 有 heldout 测试，NewtonBench 有真实物理定律），所以可以构造 C 组那样的"定量评分反馈"。但在真实的科学发现场景中——比如发现一个新的物理定律、新的药物靶点、新的材料性质——**没有人知道正确答案**，那 C 组的优势还能复现吗？

### 7.1 核心矛盾

```
实验结论:  密集定量反馈 (C) >>> 其他策略
真实世界:  没有 ground truth → 无法计算 R² → 无法给出定量反馈
                                         → C 组的优势看似不可用？
```

### 7.2 但现实中存在"不需要 ground truth 的定量反馈"

关键认识：**定量反馈不一定要依赖 ground truth**。以下是几种无需知道"正确答案"也能构造定量验证信号的途径：

#### (a) 内部一致性检验（Self-Consistency）

| 方法 | 原理 | 示例 |
|------|------|------|
| **交叉验证** | 假设在训练数据子集上拟合，在剩余子集上预测 | Agent 用 5 次实验拟合公式，在第 6 次实验上预测→计算预测误差 |
| **留一法** | 每次留出一个观测点，看假设能否泛化 | 不需要 ground truth，只需要假设的**预测一致性** |
| **Bootstrap** | 多次重采样评估假设的稳定性 | 如果假设真的对，它在不同子集上应该给出一致的参数估计 |

这本质上就是科研中常用的统计方法——你不知道真值，但你可以看**假设对已有数据的泛化能力**。

#### (b) 物理/领域约束验证

| 约束类型 | 不需要 ground truth 就能检查 | 示例 |
|----------|---------------------------|------|
| **量纲分析** | 公式左右两边的物理量纲是否匹配 | F = m·v² 量纲不对（应该是 F = m·a） |
| **极限行为** | 在边界条件下是否给出合理值 | 温度为 0 时衰变率应为 0 |
| **守恒定律** | 是否满足已知的守恒律 | 能量守恒、动量守恒 |
| **单调性/对称性** | 已知的定性关系是否满足 | 距离增大力应减小 |

这些检查相当于构造了一个**不依赖真值的打分系统**：你的假设违反了几条物理约束？违反的程度有多大？

#### (c) 预测-实验闭环（最关键的现实策略）

这是真实科研中最强大的反馈形式：

```
  ┌──────────────────────────────────────────────┐
  │  1. Agent 提出假设 H                          │
  │  2. 根据 H 做出预测: "如果 x=5, 则 y 应=12.3" │
  │  3. 实际做实验: 设定 x=5, 观测到 y=11.8       │
  │  4. 计算预测误差: |12.3-11.8|/11.8 = 4.2%    │ ← 这就是定量反馈！
  │  5. Agent 根据误差修正假设                     │
  └──────────────────────────────────────────────┘
```

**注意：第 4 步的"预测误差"就是一个不需要 ground truth 的定量反馈信号。** 你不需要知道真实的物理定律是什么，你只需要看假设的**预测值**和**实验观测值**之间的差距。

这和 C 组给的 R² 反馈在本质上是一样的——只是 C 组在"隐藏点"上计算 R²，而预测-实验闭环在"新实验点"上计算误差。

#### (d) 多假设竞争（类似 Bayesian Model Selection）

| 步骤 | 做法 |
|------|------|
| 1 | Agent 同时维护多个候选假设 H₁, H₂, H₃ |
| 2 | 设计一个"能区分这些假设"的实验（optimal experiment design） |
| 3 | 执行实验，看哪个假设的预测最接近观测 |
| 4 | 淘汰表现差的假设，生成新的候选 |

每一步都能产出定量信号（各假设的似然比），而完全不需要知道真实答案。

### 7.3 重新解读实验结论

实验的真正 takeaway 不是"你需要一个全知的评分者"，而是：

> **Agent 需要一个能把"我的假设好不好"这个模糊问题转化为精确数字的机制。**

在 benchmark 中这个机制是"和 ground truth 比 R²"，但在现实中可以替换为：

| 场景 | 替代的定量反馈来源 |
|------|-------------------|
| **科学发现** | 预测-实验闭环的预测误差 + 交叉验证 + 物理约束违反度 |
| **药物发现** | 分子模拟的 binding affinity 预测 vs 实际 wet-lab 结果 |
| **材料科学** | DFT 计算预测 vs 合成实验验证 |
| **软件工程** | 这个最直接——跑测试就行，测试本身就是定量反馈 |
| **数学证明** | 形式化验证器（Lean/Coq）的 type-check 结果 |

### 7.4 对 Agent 设计的实际建议

```
设计原则:  为 Agent 构建一个"无需 ground truth 的定量评分闭环"

                 ┌──────────────┐
                 │  Agent 假设   │
                 └──────┬───────┘
                        │ 产生可验证的预测
                        ▼
              ┌──────────────────┐
              │  预测 vs 观测    │ ← 定量误差信号
              └──────────────────┘
                        │
            ┌───────────┼───────────┐
            ▼           ▼           ▼
       交叉验证    领域约束检查    新实验验证
       (内部一致)   (物理合理性)   (外部验证)
            │           │           │
            └───────────┼───────────┘
                        │ 汇总为定量分数
                        ▼
              ┌──────────────────┐
              │  Agent 修正假设   │
              └──────────────────┘
```

核心思想：**C 组赢不是因为它有上帝视角，而是因为它把模糊的"好不好"变成了精确的数字。在现实中，用预测-验证闭环同样可以做到这一点。**
